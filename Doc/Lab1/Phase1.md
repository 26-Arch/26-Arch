### 🤖 Assistant

阶段一的核心目标是**“搭骨架”**。在这个阶段，我们不需要CPU能真正跑通任何复杂的计算，而是要确保流水线的物理结构（5个阶段模块 + 4个段间寄存器）被正确定义并连接起来，同时把开发环境和工具链的基调定好。
以下是阶段一的细化指导、工具使用策略以及最终的 Checklist：

### 1. 项目初始化与 `skill.sh` 的正确打开方式
在大型实验框架中，系统脚本通常用于挂载环境变量、配置仿真器（Verilator）路径或初始化测试脚手架。

- **如何执行**：打开终端，在项目根目录下运行 `source skill.sh` 或 `bash skill.sh`（具体取决于脚本内的定义）。建议使用 `source`，因为这能确保脚本中导出的环境变量直接生效于你当前的终端会话。
- **了解项目结构**：运行后，关注终端的输出信息。该脚本通常会为你解锁 `make test-lab1` 等测试命令的前置条件。你需要确保你的核心代码只在规定的 `vsrc/core.sv` （或其子模块文件）中编写，不要随意修改外围的 Difftest（差分测试）接入代码，否则会导致后续测试根本无法启动。

### 2. 基于 Cursor 的开发基础方法与“避坑”指南
在构建流水线骨架时，AI 助手是极好的效率工具，但如果不加以限制，它也会成为引发致命 Bug 的源头。

- **基础使用方法**：
  - **利用它写“体力活”代码**：流水线的模块间连接需要声明大量的 `wire` 和端口（例如把 PC 值、指令码、寄存器数据一级一级往后传）。你可以写好第一个模块的接口，然后让 Cursor 帮你自动补全其他阶段的类似接口声明。
  - **让它做翻译官**：遇到《RISC-V Reader》中晦涩的指令格式描述时，可以直接选中并在 Cursor 中提问：“请帮我解析 RV64I 中 B 型指令的立即数拼接逻辑，并给出 Verilog 组合逻辑代码片段。”
- **重点注意的点（极度危险的“坑”）**：
  1. **时序与组合逻辑混淆（打拍错误）**：这是大模型最容易犯的错。在 Verilog 中，**流水线段间寄存器**必须是时序逻辑（使用 `always @(posedge clk)` 和非阻塞赋值 `<=`）；而**各阶段内部的处理**（如 ALU 计算、译码）通常是组合逻辑（使用 `always @(*)` 和阻塞赋值 `=`）。Cursor 经常会把组合逻辑写进时钟沿触发的 block 中，导致信号无故延迟一拍（打拍错误）。
  2. **“上帝视角”连线**：大模型为了让你赶紧拿到结果，有时会直接把 IF 阶段获取的信号，跨过中间的所有寄存器，直接一根线连到 WB 阶段。这在物理上摧毁了流水线的隔离性。你必须强制要求它：“所有信号只能通过 IF/ID -> ID/EX -> EX/MEM -> MEM/WB 的寄存器一级一级向后传递”。
  3. **违禁语法**：再次强调，如果你让它写一个乘法扩展，它会毫不犹豫地写出 `assign res = a * b;`。你必须审查它的代码，确保没有直接调用 `*` 或 `/`。

### 3. 骨架搭建实操指南
在 `vsrc/` 目录下（或直接在 `core.sv` 内），你需要划分出以下两大类模块：

#### A. 组合逻辑模块（五大流水级）

- **IF (Instruction Fetch)**：包含 PC 的更新逻辑（如果是普通顺序执行就是 PC + 4）。
- **ID (Instruction Decode)**：接入寄存器堆（Register File）读取数据，切分 32 位指令机器码，提取出操作码、源/目的寄存器号、立即数。
- **EX (Execute)**：实例化 ALU，根据 ID 传来的控制信号进行加减逻辑运算。
- **MEM (Memory)**：负责与内存总线交互（Lab1 中主要预留接口，处理 Load/Store 的地址与数据）。
- **WB (Write Back)**：决定最终写回寄存器堆的数据是来自 ALU 的计算结果，还是来自 MEM 读取的内存数据。

#### B. 时序逻辑模块（四大段间寄存器）

- **IF_ID_Reg**、**ID_EX_Reg**、**EX_MEM_Reg**、**MEM_WB_Reg**。
- 它们的作用只有一个：在每个时钟上升沿（`posedge clk`），将前一级计算出的所有信号（包括数据、控制信号、PC 值）原封不动地锁存，并在下一个周期提供给后一级。
